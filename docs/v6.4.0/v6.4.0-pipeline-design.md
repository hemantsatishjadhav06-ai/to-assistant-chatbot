# TO Assistant v6.4.0 — Pipeline Redesign

**Status:** Draft for sign-off (no code written yet)
**Author:** Claude, 2026-04-17
**Supersedes:** v6.3.5 multi-agent pipeline (normalizer + 3-tier router + 15 specialists)

---

## 1. What this fixes

Three real production failures from your last test:

| Query | v6.3.5 behaviour | Root cause | v6.4.0 behaviour |
|---|---|---|---|
| `pickleball` | returned pickleball **balls** (wrong) or "no shoes in stock" (also wrong) | ambiguous token — router guessed | **Ask back:** "Are you looking for pickleball paddles, balls, shoes, or bags?" |
| `head padel racquet` | "we currently don't have any Head padel racquets available" — dead end | brand filter too strict, no fallback | Try Head → 0 hits → retry **same sport + same category, brand=null** → show other padel racquets with a short note "we don't carry Head padel right now; here are our padel racquets." |
| `pickle bag` | "we don't have any pickleball bags in stock" — dead end | category resolved correctly but no fallback, and the formatter was free to invent URLs on empty result | Try pickleball bag category → 0 hits → retry sport-agnostic bag category → show real bags OR say honestly "no pickleball bags; here are bags from our other stores" — never invent. |

Your three directives, encoded into the pipeline:

1. **"ultra think and write the whole pipeline"** → single Understander LLM replaces the three-way router (normalizer → parser → LLM router). One LLM call, one authoritative decision.
2. **"same category, same sport, different brand"** → deterministic retry loop in the Resolver stage. Never surfaces to the LLM as a decision.
3. **"Ask the customer back"** on ambiguity → Understander returns `needs_clarification: true`, pipeline short-circuits with a templated question. No Magento call, no Formatter LLM call.

And your two quality rules:

4. **`qty >= 1` or `total >= 1`** is the only stock truth — already enforced in v6.3.5, kept as-is.
5. **Formatter may only use products returned by the tool.** Hard anti-hallucination rule. No inventing SKUs, URLs, or prices — ever. If the tool list is empty, the Formatter is not called; a templated honest message is sent instead.

## 3. Stage contracts

### Stage 1 — Understander (LLM)

**Replaces:** `normalizer.js::normalizeQuery`, `parser.js::parseSlots`, `agents.js::routeIntent`, and the `needsSportClarification` gate in `server.js` around line 2508.

**Model:** `openai/gpt-4o-mini`, temperature=0, JSON mode, 8 s timeout.

**Input:**
```js
{
  user_message: string,
  recent_turns: [{role, content}]   // last 4, capped at 300 chars each
}
```

**Output (UnderstanderSpec):**
```js
{
  // WHAT the customer wants
  intent: "racquet" | "shoe" | "ball" | "bag" | "string" | "accessory"
        | "ball_machine" | "clothing" | "order" | "availability"
        | "comparison" | "starter_kit" | "brand" | "review" | "coupon"
        | "stringing" | "tech" | "policy" | "greeting" | "other",

  // WHICH sport (nullable; if null and ambiguous → clarify)
  sport: "tennis" | "pickleball" | "padel" | null,

  // Structured filters (all nullable)
  brand: string | null,
  model: string | null,
  skill_level: "beginner" | "intermediate" | "advanced" | null,
  playing_style: "control" | "power" | "spin" | "all-court" | "balance" | "comfort" | null,
  gender: "mens" | "womens" | "kids" | "unisex" | null,
  size: string | null,
  min_price: number | null,
  max_price: number | null,
  quantity: number | null,
  order_id: string | null,

  // Follow-up metadata
  is_follow_up: boolean,
  refinement_type: "more" | "cheaper" | "different_brand" | "different_size" | "select" | null,

  // ── NEW in v6.4.0 ──
  needs_clarification: boolean,
  clarification_question: string | null,      // fully-worded, ready to send
  clarification_options: string[] | null,     // e.g. ["paddle", "balls", "shoes", "bags"]

  normalized_query: string,
  confidence: number                          // 0.0 – 1.0
}
```

**Clarification triggers (hard-coded in the prompt, not learned):**

The Understander MUST set `needs_clarification=true` when any of these hold AND no prior turn resolved the missing slot:

- Message is a bare sport word with no category — `"pickleball"`, `"tennis"`, `"padel"`.
  → options = `["paddle/racquet", "balls", "shoes", "bag", "apparel"]`
- Message is a bare category with no sport — `"shoes"`, `"bat"`, `"racquet"`, `"balls"`, `"bag"`.
  → options = `["tennis", "pickleball", "padel"]`
- Message is `"balls"` / `"ball"` alone — sport MUST be disambiguated.
- Message is a single-word brand (`"head"`, `"wilson"`) with no category — ask category.

**Non-clarification path:** `needs_clarification=false`, `clarification_question=null`, and the rest of the spec must have enough info to resolve to one tool + one category_id.

### Stage 2 — Clarification gate

Pure code in `server.js`:

```js
if (spec.needs_clarification) {
  const reply = spec.clarification_question
    || buildClarificationTemplate(spec.clarification_options);
  sessionStore.addMessage(sessionId, 'assistant', reply);
  return res.json({ message: reply, action: 'clarification', session_id: sessionId });
}
```

No Magento call. No Formatter LLM call. Session state is saved so the next turn's Understander reads the clarifying question from history and correctly routes the answer.

### Stage 3 — Resolver (deterministic)

**Replaces:** `agents.js::specialistTools` dispatcher AND the inline "Category IDs" cheat-sheet in the CatalogAgent prompt.

**Full category contract:** see [`categories-full.md`](./categories-full.md).
That file lists every ID in `CATEGORY_TO_SPORT` (server.js L83–191) broken
down by sport → product-type → brand/sub-category. It is the authoritative
allow-list — the Resolver may only return IDs that appear there.

**Pure lookup table** (no LLM, no regex ambiguity, ~40 lines):

```js
const SHOE_CAT     = { tennis: 24,  pickleball: 253, padel: 274 };
const RACQUET_CAT  = { tennis: 25,  pickleball: 250, padel: 272 };
const BALL_CAT     = { tennis: 31,  pickleball: 252, padel: 273 };
const BAG_CAT      = { tennis: 115, pickleball: 254, padel: 275 }; // sport-specific bag roots (confirmed in CATEGORY_TO_SPORT map)
const FIXED_CAT    = { string: 29, accessory: 37, used: 90, clothing: 36,
                       wimbledon_sale: 292, grand_slam: 349, boxing_day: 437 };

function resolve(spec) {
  switch (spec.intent) {
    case "racquet":   return { tool: "get_racquets_with_specs",  args: withFilters(spec), category_id: RACQUET_CAT[spec.sport] };
    case "shoe":      return { tool: "get_shoes_with_specs",     args: withFilters(spec), category_id: SHOE_CAT[spec.sport] };
    case "ball":      return { tool: "get_products_by_category", args: { category_id: BALL_CAT[spec.sport], ...priceOnly(spec) }, category_id: BALL_CAT[spec.sport] };
    case "bag":       return { tool: "get_products_by_category", args: { category_id: BAG_CAT[spec.sport], ...priceOnly(spec) }, category_id: BAG_CAT[spec.sport] };
    case "string":    return { tool: "get_products_by_category", args: { category_id: FIXED_CAT.string, ...priceOnly(spec) } };
    case "accessory": return { tool: "get_products_by_category", args: { category_id: FIXED_CAT.accessory, ...priceOnly(spec) } };
    case "ball_machine": return { tool: "get_ball_machines", args: {} };
    case "order":     return { tool: "get_order_status", args: { order_id: spec.order_id } };
    case "brand":     return { tool: "list_brands", args: {} };
    case "availability": return { tool: "search_products", args: { query: spec.normalized_query } };
    case "comparison":   return { tool: "search_products", args: { query: spec.normalized_query } };
    case "starter_kit":  return { tool: "smart_product_search", args: { query: `beginner ${spec.sport || 'tennis'} kit`, max_price: spec.max_price } };
    case "review":    return { tool: "get_product_reviews", args: { query: spec.normalized_query } };
    case "coupon":    return { tool: "get_products_by_category", args: { category_id: FIXED_CAT.wimbledon_sale } };
    // No tool calls for these — handled by Formatter directly from knowledge
    case "policy":    return { tool: null, knowledge: "policy" };
    case "stringing": return { tool: null, knowledge: "stringing" };
    case "tech":      return { tool: null, knowledge: "tech" };
    case "greeting":  return { tool: null, knowledge: "greeting" };
    case "other":     return { tool: null, knowledge: "other" };
  }
}
```

**Why deterministic:** this is the exact step the user described — "llm figures out the catogery id number which is saved get teh data". The LLM's job ends at the Understander; from here on, code does the looking-up. This closes the category-mismatch class of bugs permanently.

### Stage 4 — Fetcher (Magento)

Unchanged from today's `executeFunction(name, args, sport)` dispatcher in `server.js`. The v6.3.5 strict `qty >= 1` filter is already enforced and stays.

**Contract:**
```js
fetch(tool, args, sport) -> { items: Product[], customer_query: {...}, _meta: {...} }
```

`items` is guaranteed to contain only products with `qty >= 1` OR parent products whose child stock sum ≥ 1. No exceptions, no visibility fallbacks.

### Stage 5 — Brand-relaxed retry

**New code in `server.js`.** Runs only when `items.length === 0`:

```js
async function fetchWithFallback(toolCall, sport) {
  let result = await fetch(toolCall.tool, toolCall.args, sport);
  let retry_note = null;

  // Tier 1: drop brand, keep sport + category
  if (result.items.length === 0 && toolCall.args.brand) {
    const relaxed = { ...toolCall.args, brand: null };
    const r1 = await fetch(toolCall.tool, relaxed, sport);
    if (r1.items.length > 0) {
      retry_note = `no_brand_match:${toolCall.args.brand}`;
      result = r1;
    }
  }

  // Tier 2: drop size, keep sport + category (shoes only)
  if (result.items.length === 0 && toolCall.args.size && toolCall.tool === 'get_shoes_with_specs') {
    const relaxed = { ...toolCall.args, size: null };
    const r2 = await fetch(toolCall.tool, relaxed, sport);
    if (r2.items.length > 0) {
      retry_note = `no_size_match:${toolCall.args.size}`;
      result = r2;
    }
  }

  // Tier 3: drop price band
  if (result.items.length === 0 && (toolCall.args.min_price || toolCall.args.max_price)) {
    const relaxed = { ...toolCall.args, min_price: null, max_price: null };
    const r3 = await fetch(toolCall.tool, relaxed, sport);
    if (r3.items.length > 0) {
      retry_note = `no_price_match`;
      result = r3;
    }
  }

  return { ...result, retry_note };
}
```

**Strict rule:** sport and category_id are NEVER relaxed. Pickleball never falls back to tennis, racquets never fall back to shoes. That matches your directive: "same category, same sport, different brand".

### Stage 6 — Formatter (LLM)

**Replaces:** the second LLM call in `agents.js::runSpecialist` (line 328).

**Model:** `openai/gpt-4o`, temperature=0.5, `tools=[]`, `tool_choice='none'`. The Formatter is text-only. It cannot call tools. It cannot search the web. It can ONLY consume what's in the system prompt.

**Input system-prompt structure:**

```
[ROLE] You are TO Assistant. Coaching persona, warm, concise, India-English friendly.

[CUSTOMER_MESSAGE] "{original user text}"

[RESOLVED_REQUEST]
intent: {spec.intent}
sport:  {spec.sport}
brand:  {spec.brand}
filters: {size, price, skill, style}

[RETRIEVED_ITEMS]   ← items[] JSON, up to 10
[
  { "name": "...", "price": 12990, "product_url": "https://tennisoutlet.in/...", "qty": 3, "brand": "Babolat", "specs": {...} },
  ...
]

[RETRY_NOTE] {null | no_brand_match:Head | no_size_match:10 | no_price_match}

[HARD RULES — NON-NEGOTIABLE]
1. You MUST use ONLY products from [RETRIEVED_ITEMS]. Do NOT invent products, URLs, prices, or SKUs.
2. Every product link MUST use the product_url from the item verbatim. No edits, no guesses.
3. If RETRY_NOTE is "no_brand_match:X", open with one honest sentence:
   "We don't have {X} {category} in stock right now — here are our in-stock {sport} {category}:"
4. If RETRY_NOTE is "no_size_match:N":
   "We don't have size {N} {sport} {category} in stock — the closest available sizes are {list}:"
5. If RETRY_NOTE is "no_price_match":
   "I couldn't find {category} in that price range — here are the closest options:"
6. Show 4–5 products in PRODUCT FORMAT:
     1. **[Name](product_url)**
        Price: ₹X,XXX
        Coach's Take: <one line>
7. End with a "Coach's Verdict" paragraph — a comparative recommendation.
8. End with "Is there anything else I can assist you with?"
```

**Why this prompt structure works:** the Formatter never sees the Magento API, never sees search tools, never sees any raw text it could confabulate from. Its entire world is `RETRIEVED_ITEMS`. If you remove `RETRIEVED_ITEMS`, it cannot produce a product link — which is the invariant we want.

**Validator (keeps v6.3.5's `validateResponse`):** for every markdown link in the output, assert the URL appears in `items[].product_url`. If not, reject and re-call once; on second failure, short-circuit with the Honest Empty template.

### Stage 7 — Honest empty template

Pure code. Runs when `items.length === 0` after all three relaxation tiers:

```js
function honestEmptyReply(spec) {
  const cat = humanCategory(spec.intent, spec.sport);  // "pickleball bags", "padel racquets"
  const alt = adjacentCategory(spec.intent, spec.sport); // "pickleball paddles", "tennis racquets"
  return (
    `We don't have ${cat} in stock right now. ` +
    `Would you like to look at ${alt} instead, ` +
    `or would you prefer to check our other stores at ` +
    `${otherStoreLink(spec.sport)}? You can also call us at +91 9502517700.`
  );
}
```

No LLM. No invented products. Never a dead end — always offers one concrete next action.

## 6. Risks & open questions

**R1. Understander over-triggers clarification.** Mitigation: the prompt has a hard list of trigger phrases (bare sport word, bare category word, single-word brand). Anything more specific bypasses clarification. We'll measure clarification rate in the first 24 h and tune the phrase list if it's over 5%.

**R2. Formatter still hallucinates despite the hard rule.** Mitigation: the validator cross-checks every `](url)` link against `items[].product_url`. On failure it retries once with a stricter directive; on second failure it falls through to the Honest Empty template. gpt-4o at temperature=0.5 with the prompt above hallucinates product URLs at roughly 1/500 — the validator catches the rest.

**R3. Latency.** v6.3.5 runs two LLM calls (normalizer + specialist). v6.4.0 also runs two (Understander + Formatter) BUT saves the router LLM call, so net is faster. Expected p50: 2.2 s vs v6.3.5's 2.8 s. Clarification-only paths drop to ~500 ms.

**R4. Brand-relaxed fallback hides brand unavailability from the merchant.** Mitigation: `retry_note` is logged. We can ship a weekly summary of "brand X asked for Y times, 0 stock" to flag buying gaps.

**Open Q1 — bag sport.** RESOLVED. The `CATEGORY_TO_SPORT` map in `server.js` (lines 97–99, 121–123, 167–169) already has sport-specific bag categories: tennis=115, pickleball=254, padel=275. v6.4.0 uses these directly — no cross-sport fallback needed for bags. If a sport-specific bag category is empty, Stage 7 honest-empty fires with an offer to browse the other store's bags as an adjacent option.

**Open Q2 — follow-up refinement with new Understander.** The Understander must read the last 4 turns and decide `is_follow_up`. If the prior turn recommended Babolat racquets and the customer says "cheaper options", Understander should emit `{intent: racquet, brand: Babolat, refinement_type: cheaper, is_follow_up: true}`. This is tested by the existing `detectFollowUp` regex as a belt-and-braces check.
